## Purpose

This project is being built as a long-lived open source foundation.
Contributions should strengthen clarity, maintainability, and architectural discipline.

## Before You Start

- Read `README.md`, `foundation/` documents, and the architecture pages.
- Confirm the change fits the current scope and non-goals.
- Prefer discussion and ADRs for changes that affect boundaries or terminology.

## Proposed Workflow

1. Open an issue or discussion when the change is non-trivial.
2. Write or update the relevant design document first.
3. Keep changes focused and easy to review.
4. Avoid adding implementation before the foundation is agreed.

## Review Expectations

- Changes should remain engine-agnostic.
- Public concepts should be named consistently across docs.
- Architectural choices should be justified in an ADR when they have lasting impact.

## Documentation Standards

- Use clear headings and an index in every major document.
- Prefer short, direct sections over long narrative blocks.
- Keep terminology stable across the repository.
